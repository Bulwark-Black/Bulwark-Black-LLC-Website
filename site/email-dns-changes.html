Why: Resend sends with the Return-Path on the send. subdomain (you already have that SPF set correctly). The root domain itself does not send mail, so publishing an empty SPF (-all) tells the world that any mail with MAIL FROM = ruraltechandsupport.com is forged. This blocks spoofing at the SPF layer and is fully compatible with how Resend is already working (DMARC will still pass via DKIM alignment).

Skip this if: you plan to add a mailbox or sending service at the root domain later. In that case leave it off until you know what to allow.

2. bulwarkblack.com (the receive side, separate zone)

Form notifications land in support@bulwarkblack.com, which is on Microsoft 365 (Outlook). Receive works, but three DNS records on this domain are wrong or missing. None of these affect form receive today, but they do prevent you from reliably sending mail FROM bulwarkblack.com via Outlook (responses, replies, anything client-facing). Worth fixing now.

replace  1 Fix the SPF to point at Microsoft 365 instead of GoDaddy

Find and edit the existing TXT record on @ that contains v=spf1 include:secureserver.net -all and change its data to:

Type
TXT
Name
@
Data
v=spf1 include:spf.protection.outlook.com -all

Why: Your MX is bulwarkblack-com.mail.protection.outlook.com (Microsoft 365), but the SPF still says GoDaddy sends mail for you. Outlook outbound mail currently fails SPF as a hardfail (the -all), which means recipients are likely sending your replies to spam.

If GoDaddy hosting also sends mail (forms on the GoDaddy site, for example), use this combined version instead: v=spf1 include:spf.protection.outlook.com include:secureserver.net -all

add  2 Publish a DMARC policy
Type
TXT
Name
_dmarc
Data
v=DMARC1; p=none; rua=mailto:dmarc@bulwarkblack.com; pct=100; adkim=r; aspf=r

Why: Currently there is no DMARC record on bulwarkblack.com. Start with p=none (monitor only) for a week or two, watch the reports that land in dmarc@bulwarkblack.com (set up that mailbox or forward to support), then tighten to p=quarantine once you confirm only legitimate sources are signing your mail.

add  3 Enable Microsoft 365 DKIM and publish the two CNAMEs

This is a two-step process. First, enable DKIM in the Microsoft 365 admin center, then publish the CNAMEs at GoDaddy.

Microsoft has already given you the exact CNAME targets. Add these two CNAMEs at GoDaddy DNS for bulwarkblack.com first, then go back to the M365 DKIM page and click Enable.

Watch out: the target ends in .microsoft (a real top-level domain Microsoft owns), not .com. Paste it exactly as written. Do not add a trailing dot in GoDaddy's UI; GoDaddy treats CNAME targets as fully qualified automatically.

After saving, open a terminal and run dig +short selector1._domainkey.bulwarkblack.com CNAME to confirm the record resolves. Microsoft says sync can take up to 4 days but GoDaddy is usually live within a few minutes. Once both resolve, return to https://security.microsoft.com → Policies & rules → Threat policies → Email authentication settings → DKIM and click "Enable" for bulwarkblack.com.

Order of operations

  1. Clean up ruraltechandsupport.com (the four deletions). Safe to do first, no service impact.
  2. Optionally add the root SPF (v=spf1 -all at @).
  3. Replace the bulwarkblack.com SPF record.
  4. Publish DMARC for bulwarkblack.com.
  5. Enable M365 DKIM and publish the two selector CNAMEs.
  6. Wait an hour, then test:

    • Send a test message from support@bulwarkblack.com to a Gmail account, view headers, confirm SPF: pass, DKIM: pass, DMARC: pass.
