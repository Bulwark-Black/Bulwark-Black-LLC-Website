<p>In a stark demonstration of how artificial intelligence is transforming the cybersecurity threat landscape, the Sysdig Threat Research Team (TRT) has documented a sophisticated cloud intrusion where attackers achieved full administrative control of an AWS environment in <strong>less than 10 minutes</strong> — with strong evidence that large language models (LLMs) were used to automate the entire attack chain.</p>
<p>The incident, observed on November 28, 2025, represents a significant escalation in the speed and sophistication of cloud attacks, highlighting the urgent need for organizations to implement automated defenses capable of matching the pace of AI-driven offensive operations.</p>
<h2>The Attack Chain: From Exposed Credentials to Full Compromise</h2>
<h3>Initial Access: Hunting AI Infrastructure</h3>
<p>The attack began when threat actors discovered valid credentials stored in public S3 buckets. Notably, these buckets were named using common AI terminology — suggesting the attackers were specifically targeting organizations with AI/ML infrastructure. The compromised credentials belonged to an IAM user with permissions on AWS Lambda and Amazon Bedrock, likely created to automate AI tasks.</p>
<h3>Privilege Escalation: Lambda Code Injection</h3>
<p>After initial reconnaissance across multiple AWS services (Secrets Manager, SSM, EC2, ECS, RDS, and more), the attackers exploited Lambda permissions to escalate privileges. They modified an existing Lambda function called <code>EC2-init</code>, injecting Python code that:</p>
<ul>
<li>Listed all IAM users and their access keys</li>
<li>Created new administrative access keys for a privileged user</li>
<li>Enumerated S3 buckets and their contents</li>
</ul>
<p><strong>The malicious code contained comments written in Serbian</strong>, potentially indicating the threat actor&#8217;s origin. More significantly, the code&#8217;s structure — including comprehensive exception handling and the speed of iteration — strongly suggests it was generated by an LLM.</p>
<h3>Evidence of AI Hallucinations</h3>
<p>Multiple indicators point to LLM-assisted attack automation:</p>
<ul>
<li><strong>Hallucinated AWS account IDs:</strong> The attackers attempted to assume roles in account IDs like <code>123456789012</code> and <code>210987654321</code> — sequential test numbers that don&#8217;t exist in production environments</li>
<li><strong>Non-existent GitHub repositories:</strong> Scripts referenced <code>https://github.com/anthropic/training-scripts.git</code>, a repository that doesn&#8217;t exist — a classic LLM hallucination</li>
<li><strong>Session naming:</strong> Role assumption attempts used revealing session names like &#8220;explore,&#8221; &#8220;pwned,&#8221; &#8220;escalation,&#8221; and notably &#8220;claude-session&#8221;</li>
</ul>
<h2>LLMjacking: Stealing AI Compute</h2>
<p>With administrative access secured, the attackers pivoted to what Sysdig calls &#8220;LLMjacking&#8221; — abusing Amazon Bedrock to invoke multiple AI models at the victim&#8217;s expense. They invoked high-end models including:</p>
<ul>
<li>Claude Sonnet 4 and Claude Opus 4</li>
<li>Claude 3.5 Sonnet and Claude 3 Haiku</li>
<li>DeepSeek R1</li>
<li>Llama 4 Scout</li>
<li>Amazon Nova Premier and Titan Image Generator</li>
</ul>
<p>The attackers also created a Terraform module designed to deploy a backdoor Lambda function that would generate Bedrock credentials via a publicly accessible URL — requiring no authentication.</p>
<h2>GPU Instance Hijacking: $23,600/Month at Stake</h2>
<p>The threat actors attempted to launch p5.48xlarge GPU instances (named &#8220;stevan-gpu-monster&#8221;) for apparent model training. When these failed due to capacity constraints, they successfully launched a <strong>p4d.24xlarge instance</strong> — which would cost approximately <strong>$23,600 per month</strong> if left running.</p>
<p>A JupyterLab server was configured to start automatically, providing persistent backdoor access to expensive compute resources even if AWS credentials were revoked.</p>
<h2>Critical Defensive Recommendations</h2>
<p>Sysdig TRT recommends these essential mitigations:</p>
<h3>1. Eliminate Long-Term Credentials</h3>
<p>Avoid IAM users with static access keys. Use IAM roles with temporary credentials whenever possible. If long-term credentials are required, implement strict rotation policies.</p>
<h3>2. Secure AI-Related Infrastructure</h3>
<p>S3 buckets containing AI data, models, or code should never be publicly accessible. Attackers are actively hunting for AI-related resources.</p>
<h3>3. Apply Least Privilege to Lambda</h3>
<p>Developers should not have <code>UpdateFunctionCode</code> permissions unless absolutely necessary. This permission enables privilege escalation through code injection.</p>
<h3>4. Monitor Bedrock Usage</h3>
<p>Create alerts for invocation of AI models not normally used by your organization. Consider implementing Service Control Policies (SCPs) to restrict which models can be invoked.</p>
<h3>5. Implement Real-Time Cloud Detection</h3>
<p>With attackers achieving admin access in under 10 minutes, organizations must deploy automated detection capable of identifying and responding to suspicious activity in real-time.</p>
<h2>The Bottom Line</h2>
<p>This incident signals a fundamental shift in cloud security. By automating the attack chain with AI, threat actors have dramatically reduced &#8220;dwell time&#8221; — the window defenders have to detect and respond. Organizations must prioritize automated, real-time detection to match this new pace of exploitation.</p>
<p>The era of &#8220;offensive AI&#8221; agents that can execute complex intrusions faster than human defenders can react is no longer theoretical — it&#8217;s here.</p>
